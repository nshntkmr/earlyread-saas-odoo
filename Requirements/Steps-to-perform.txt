2.3 Multi-Page Navigation Architecture

Navigation between pages does NOT reload the browser. The SidebarNav
component updates a reactive state variable (currentPageKey), which
triggers the TabBar and TabContent to re-render with the new page\'s
configuration. This is a single-page application pattern within the Odoo
portal.

-   Each sidebar item maps to a dashboard.page record (e.g.,
    key=\"overview\", key=\"hospitals\")

-   Clicking a sidebar item calls this.state.currentPageKey =
    \"hospitals\"

-   The TabBar component reacts to the page change and loads tabs from
    the config model

-   The TabContent component lazy-loads only the active tab\'s widgets

-   URL updates via history.pushState for bookmarkable deep links:
    /my/practice-vantage/hospitals/performance

-   Browser back/forward works via popstate listener in the root
    component

4.2 Click-Through Drill-Down System

Every widget supports two types of click actions, configured per widget
via the config model:

**Type 1: Internal Navigation (page-to-page)**

The widget fires an event that the root component handles by updating
currentPageKey and optionally currentTabKey. Example: clicking the
Market Share stat tile navigates to the Competitive Intel page, Market
Share tab.

**Type 2: External Navigation (to a detail URL)**

The widget navigates to a new portal route with the clicked entity\'s
ID. Example: clicking a physician name in a data table navigates to
/my/practice-vantage/physician/1234. The detail page is a separate
portal controller with its own Owl components.

**Type 3: Filter Application (chart element click)**

Clicking a bar, slice, or map region applies a filter to all widgets on
the current tab. Example: clicking the \"PHY\" bar in Source Mix adds
source_type=PHY to the global filter state, and all other widgets
re-render with this filter applied. This is implemented via a shared
FilterState service injected into all widget components.

5\. AI Insights System

The \"Are We Winning?\" banner and other AI-generated text tiles are
powered by a server-side insight generation pipeline. Insights are NOT
generated on every page load --- they are pre-computed and cached.

5.1 How AI Insights Work

1.  A nightly cron job (ir.cron) runs for each practice with active AI
    insight widgets.

2.  For each widget, the job queries the relevant materialized view and
    assembles key metrics into a structured context.

3.  A prompt template (stored in
    dashboard.widget.config.prompt_template) is filled with the metrics.
    Example: \"Analyze this HHA\'s Q4 performance: {admits} admits
    ({admits_yoy_pct}% YoY), {denial_rate}% denial rate (target:
    {denial_target}%), timely access at {timely_access_pct}%. In 2-3
    sentences, identify the most critical issue and its business
    impact.\"

4.  The completed prompt is sent to OpenAI GPT-4 (via API) or Odoo 19\'s
    native ai.agent model (Enterprise).

5.  The response is stored in dashboard.ai.insight with a 24-hour TTL
    and a hash of the input data.

6.  On page load, the AIInsightTile component fetches the cached
    insight. If the cache is stale (data changed since last generation),
    a background refresh is triggered.

6\. Admin-Editable Layout System

The admin layout editor uses gridstack.js to allow non-technical
administrators to rearrange, resize, add, and remove dashboard widgets
from the browser. This section covers how it integrates with the Odoo 19
Owl framework.

6.1 How the Layout Editor Works

1.  Admin logs into the backend and navigates to a dashboard page.

2.  An \"Edit Layout\" toggle button appears (visible only to users in
    group_dashboard_admin).

3.  Clicking it enables gridstack\'s drag-and-drop mode. Widgets show
    resize handles and drag handles.

4.  Admin can drag widgets to new positions, resize them by dragging
    edges, or remove them via an \"×\" button.

5.  An \"Add Widget\" button opens a modal with the widget picker:
    choose type, data source, and initial config.

6.  Clicking \"Save Layout\" serializes the gridstack state to JSON and
    saves it to dashboard.layout.

7.  Portal users always see the saved layout (gridstack in static/locked
    mode).

7\. Data Pipeline: CSV to Dashboard

This section covers the complete data flow from raw CSV files to
rendered dashboard widgets. The pipeline runs once for initial load and
then incrementally for updates.

7.1 Initial Load Pipeline

1.  Stage CSVs to an UNLOGGED table using PostgreSQL COPY command (\~3-5
    minutes for 25M rows).

2.  Cast and insert into the partitioned healthcare_claims table
    (partitioned by month).

3.  Create BRIN indexes on date columns and B-tree indexes on
    categorical filters.

4.  Create materialized views (mv_claims_monthly_summary,
    mv_provider_performance, mv_daily_trends, etc.).

5.  Create unique indexes on materialized views to enable CONCURRENT
    refresh.

6.  Register unmanaged Odoo models (\_auto = False) pointing at each
    materialized view.

7.  Verify Odoo can read the models via the backend form/list views.

8.  Configure ir.cron jobs to refresh materialized views on schedule.

7.2 Incremental Update Pipeline

When new CSV files arrive (daily or weekly):

1.  New CSVs are staged to the UNLOGGED table.

2.  An upsert operation (INSERT \... ON CONFLICT) merges new/updated
    claims into healthcare_claims.

3.  Materialized views are refreshed concurrently (non-blocking) via
    pg_cron or Odoo ir.cron.

4.  Dashboard.ai.insight records are flagged as stale (prompt_hash no
    longer matches current data).

5.  The nightly AI cron regenerates stale insights.

8.2 Security: Multi-Tenancy Record Rules

Every data model has a record rule ensuring practices can only see their
own data:

-   Rule name: \"Practice Own Data\"

-   Domain: \[(\"partner_id\", \"=\", user.partner_id.id)\]

-   Groups: group_practice_portal_user

-   Applied to: All healthcare.\* models

Regional managers get a broader rule: \[(\"region_id\", \"in\",
user.partner_id.region_ids.ids)\]. Healthcare admins (internal) have no
domain restriction.
